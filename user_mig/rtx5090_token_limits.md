# RTX 5090 + Gemma 4 31B 토큰 한계 현황

> Nexus 개발 과정에서 확인된 RTX 5090 (32GB) 환경의 실측 제약 사항을 정리한다.
> 다른 AI 도구(Claude, GPT, Gemini 등)와 협의 시 공유 자료로 활용한다.

---

## 1. VRAM 제약

| 항목 | 수치 |
|---|---|
| RTX 5090 VRAM | 32GB |
| Gemma 4 31B (INT4 양자화) | ~20GB |
| vLLM 서빙 오버헤드 | ~11GB (gpu-memory-utilization 0.90) |
| 잔여 VRAM | ~1GB |
| **최대 max-model-len** | **8,192 (성공)** |
| 16,384 시도 | OOM 실패 |

## 3. 실제 사용 시 한계

| 작업 | 가능 여부 | 이유 |
|---|---|---|
| 짧은 질문 → 짧은 답변 | ✅ 가능 | 1턴에 ~2,000 토큰이면 충분 |
| 파일 1개 읽고 분석 | ⚠️ 제한적 | 파일 내용이 ~1,500자 이내여야 함 |
| 도구 1회 호출 후 응답 | ✅ 가능 | LS, Grep 등 결과가 짧은 도구 |
| 도구 2회 이상 연속 호출 | ⚠️ 위험 | 턴 누적으로 3턴째 컨텍스트 초과 |
| 큰 파일 읽기 (HTML 등) | ❌ 불가 | 파일 내용만으로 5,000+ 토큰 초과 |
| 멀티턴 대화 (5턴 이상) | ❌ 불가 | 히스토리 누적으로 컨텍스트 포화 |
| 프로젝트 전체 분석 | ❌ 불가 | 파일 수십 개를 읽을 여유 없음 |
| 24개 도구 전체 활성화 | ❌ 불가 | 스키마만 6,102 토큰 (75% 점유) |

## 4. 발생하는 구체적 문제

### 4-1. 도구 호출 실패

모델이 도구 스키마를 이해할 컨텍스트가 부족하여 빈 입력 `{}`으로 도구를 호출하는 현상이 발생한다.
예: `Read {}` (file_path 누락), `LS {}` (path 누락)

### 4-2. 컨텍스트 초과 무한 재시도

도구 결과가 messages에 누적 → 컨텍스트 초과 → vLLM 400 에러 → 재시도 → 다시 초과의 루프가 발생한다.
(CONTEXT_OVERFLOW 즉시 종료 로직으로 수정 완료)

### 4-3. 대화 단절

2~3턴 후 컨텍스트가 꽉 차서 이전 대화 맥락을 완전히 잃는다.
사용자 입장에서는 AI가 방금 한 대화를 기억하지 못하는 것처럼 보인다.

### 4-4. 문서 분석 한계

PDF/DOCX 청크 1개(~2,500자, A4 1~2페이지)만 분석 가능하다.
청크 간 맥락 연결이 불가능하여, 앞부분 내용을 참조한 뒷부분 분석이 되지 않는다.

## 5. 현재 적용한 완화 조치

| 조치 | 내용 | 효과 |
|---|---|---|
| CLI 도구 축소 | 24개 → 11개 | 스키마 6,102 → 3,000 토큰 절약 |
| 웹 도구 축소 | 24개 → 8개 | 스키마 6,102 → 2,051 토큰 절약 |
| 동적 max_tokens | 입력 크기 추정 후 출력 토큰 자동 조정 | 컨텍스트 초과 빈도 감소 |
| 입력 truncate | 컨텍스트 85% 초과 시 마지막 메시지 자르기 | 최소 512 토큰 출력 확보 |
| 자동 재시도 | vLLM 400 에러 시 max_tokens 축소 3회 | 경계 사례 복구 |
| 즉시 종료 | CONTEXT_OVERFLOW 감지 시 루프 즉시 중단 | 무한 루프 방지 |
| 웹 히스토리 예산 | user/assistant 텍스트만 6,000자 이내 복원 | 세션 간 맥락 부분 유지 |
| 문서 청크 분할 | 2,500자 단위 순차 분석 | 대용량 문서 분할 처리 가능 |

## 7. 시연 시 권장 시나리오

현재 환경에서 안정적으로 동작하는 범위:

1. **짧은 질문 응답**: "이 프로젝트가 뭐야?" → 시스템 프롬프트 기반 답변
2. **단일 파일 조회**: 작은 파일(~50줄) 읽기 → 내용 요약
3. **검색**: 특정 키워드로 파일 검색 (Grep, Glob) → 결과 목록
4. **디렉토리 조회**: LS로 폴더 구조 확인 → 설명
5. **문서 분석**: 짧은 PDF/DOCX 첨부 → 1청크 요약

피해야 할 시나리오:

- 큰 파일 읽기 요청
- "전체 분석해줘" 같은 광범위 요청
- 5턴 이상 이어지는 복잡한 대화
- 파일 수정/생성 후 추가 분석 (턴 누적)

## 8. 비교 참고: 클라우드 AI 서비스의 컨텍스트

| 서비스 | 컨텍스트 | Nexus 대비 |
|---|---|---|
| Claude Opus (Anthropic) | 200K 토큰 | 24배 |
| Claude Opus 1M (확장) | 1,000K 토큰 | 122배 |
| GPT-4o (OpenAI) | 128K 토큰 | 16배 |
| Gemini 1.5 Pro (Google) | 1,000K~2,000K 토큰 | 122~244배 |
| **Nexus (RTX 5090)** | **8,192 토큰** | **기준** |

> 컨텍스트 크기의 차이가 곧 한 번에 처리할 수 있는 정보량의 차이이다.
> 클라우드 서비스는 수백 개 파일을 동시에 읽고 분석할 수 있지만,
> 현재 Nexus는 짧은 파일 1~2개가 한계이다.
> 이는 모델이나 소프트웨어의 문제가 아니라 GPU VRAM에 의한 물리적 제약이다.
